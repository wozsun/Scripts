# 脚本编写规范

本规范适用于本仓库中的所有脚本，新增或修改脚本时，请优先参考本规范，保持脚本风格和交互习惯的一致性

所有脚本使用`UTF-8`编码，行尾使用`LF`

macOS脚本使用Zsh编写，可使用macOS自带系统函数，文件扩展名为`.zsh`

Windows脚本使用PowerShell编写，仅使用PowerShell 7，文件扩展名为`.ps1`

当用户需求与规范不符时，优先执行规范并提示用户，在收到明确修改要求后再进行调整

## 基本规范

- 注释与控制台输出使用中文
- 修改既有脚本时，保持原有参数、输出结构和用户习惯，除非有明确改进需求
- 对于可配置项，可以提取出来放置在脚本上方，避免硬编码，方便用户修改和维护
- 默认用户权限，避免需要使用管理员权限的操作
- 必须携带参数的脚本无参数直接执行时，不要直接报错中断，而是引导用户输入参数
- 脚本应优先保证数据安全，当脚本功能涉及破坏性操作时不可直接执行，详见下方`破坏性操作脚本规范`
- 脚本对本地文件的操作需保证原子性，中断或出错时，不得残留半成品文件
- 临时文件应放置在同一目录下，且命名以`.tmp`结尾，脚本结束后应删除临时文件
- 对于批量操作脚本，不要因为单个文件的错误而中断整个脚本，应记录错误并继续处理后续文件
- 对于需要扫描目录的脚本，需要输出阶段进度，详见下方`进度输出规范`
- 脚本编写时尽量使用函数封装逻辑，保持主流程清晰，便于维护和扩展，减少代码重复
- 添加或修改脚本时，同步修改README.md中的脚本列表和说明
- 脚本运行默认输出应保持清晰、简洁，参考下方`控制台输出规范`
- 脚本编辑完成后，需进行自测，确保功能正确，输出清晰，并且符合本规范要求

### 默认参数

- `-h`：显示帮助信息，说明脚本用途、参数和关键规则
- `-s`：默认只操作未隐藏文件和文件夹，传入此参数后包含隐藏项
- `-yes`: 对于需要用户确认的操作，传入此参数后跳过预览直接执行默认选项

## 控制台输出规范

控制台输出要避免冗长的文本和过多的细节。可以询问用户具体的输出要求，同时可以考虑提供调试模式，在调试模式下输出更详细的信息，正常模式下只输出必要的结果和状态。

控制台输出可以使用颜色来区分不同类型的信息，例如：

- `Cyan`（青色，建议参考色 `#00BCD4`）：流程性信息，例如阶段进度、当前步骤、菜单标题、用户输入提示。
- `Yellow`（黄色，建议参考色 `#FFC107`）：需要注意但不一定失败的信息，例如预览内容、待确认操作、跳过项、用户主动退出。
- `Green`（绿色，建议参考色 `#4CAF50`）：成功或安全状态，例如操作完成、无可处理项、无失败项。
- `Magenta`（洋红色，建议参考色 `#E91E63`）：已执行的破坏性操作，例如已删除、已移动、已覆盖、已重命名。
- `Red`（红色，建议参考色 `#F44336`）：错误或失败信息，例如输入无效、操作失败、异常原因。
- `DarkGray`（深灰色，建议参考色 `#757575`）：辅助信息，例如分隔线、摘要元数据、原因说明、调试或次要细节。

避免把整段长文本都染色。优先只给标题、状态、编号、关键结果或原因说明上色，保持输出可读。

脚本优先使用运行环境原生命名色，十六进制颜色用于说明期望视觉语义；只有在终端和脚本语言明确支持真彩色时才需要使用。

## 进度输出规范

使用普通文本单行动态进度条，推荐格式：

```text
[进度] 活动名称 [##############--------------] 50% 当前状态 ( 25 / 50 )
```

推荐实现要点：

- 只在百分比变化时刷新，减少输出噪音
- 使用单行刷新，避免滚屏
- 进度条结束后输出换行，避免后续日志和进度条混在一起

## 破坏性操作脚本规范

破坏性操作指任何对文件或文件夹有修改的操作，包括但不限于删除、移动、重命名、编辑内容等。除非用户明确要求脚本直接执行，否则此类脚本必须采用`先预览再选择执行`的流程：

1. 先扫描并计算默认执行计划
2. 展示预览，列出将被操作的文件或文件夹
3. 提供用户操作选项
4. 当用户选择操作后再执行

推荐菜单格式：

```text
请选择操作:
1 默认
2 手动
0 退出
```

`2 手动`根据用户要求决定是否删除

手动操作模式建议：

- 给预览项分配编号
- 支持逗号分隔的多编号输入，例如 `1,3,5`
- 输入无效时提示并要求重新输入
- 输入`0`表示跳过或取消当前手动操作项目；如果是多次选择，可以通过输入`00`表示不进行后续操作直接退出脚本
